React Router DOM

npm install react-router-dom

Связывает URL браузера с компонентами. При переходе на /about рендерится компонент About — без перезагрузки страницы.

Для чего используется

  • Создавать многостраничные SPA-приложения
  • Переключать страницы без перезагрузки браузера
  • Читать URL: путь, параметры (:id), query (?page=2)
  • Вкладывать страницы в Layout (шапка + контент)
  • Закрывать маршруты от неавторизованных
  • Перенаправлять пользователей (redirect)

Когда применять

✅Приложение имеет несколько "страниц"
✅Нужны кнопки Назад/Вперёд браузера
✅Нужно давать прямые ссылки на разделы
✅Нужна авторизация с редиректом
❌Лендинг из одной страницы — роутер не нужен

Redux Toolkit (RTK)

npm install @reduxjs/toolkit react-redux

Централизованное хранилище клиентского состояния. Данные доступны из любого компонента без пробрасывания props через дерево.

Что хранить в Redux

  • Токен авторизации и данные пользователя
  • Тема, язык, пользовательские настройки
  • Глобальные уведомления (toast/snackbar)
  • Корзина покупок, избранное
  • Всё что нужно в 2+ несвязанных местах

Что НЕ хранить в Redux

❌Значения полей формы → useState / RHF
❌Открыт ли попап/дропдаун → useState
❌Данные от сервера → RTK Query
❌Данные нужны только одному компоненту → useState

RTK Query

встроен в @reduxjs/toolkit

Запросы к API с автоматическим кешированием, инвалидацией и загрузочными состояниями. Хранит данные в том же Redux store.

Для чего используется

  • GET-запросы с кешированием результата
  • POST/PUT/DELETE мутации данных
  • Автообновление UI после мутации (теги)
  • Дедупликация одинаковых запросов
  • Polling — автообновление каждые N секунд
  • Prefetch — предзагрузка при hover

RTK Query vs Thunk

ThunkRTK Query
КешВручнуюАвто
isLoadingВручнуюАвто
ИнвалидацияВручнуюТеги
ДублиВозможныНет
КодаМногоМинимум

Кеширование и инвалидация тегов

getPosts() запрос
→
Кеш: tag Post/LIST
→
createPost() мутация
→
invalidates Post/LIST
→
getPosts() авто-рефетч
Теги — главная суперсила RTK Query. После мутации нужные данные перезапрашиваются автоматически.

Что возвращает useQuery — все поля

ПолеКогда true / что содержит
isLoadingПервый запрос, кешированных данных ещё нет
isFetchingЛюбой активный запрос — в т.ч. фоновое обновление
isSuccessПоследний запрос завершился успешно
isErrorПоследний запрос завершился ошибкой
dataДанные последнего успешного ответа
errorОбъект ошибки (status, data)
refetchФункция для ручного повторного запроса
isLoading vs isFetching: isLoading = нет данных вообще. isFetching = любой запрос в процессе, даже если кешированные данные уже есть.

Axios

npm install axios

HTTP-клиент для запросов к API. Популярнее нативного fetch в реальных проектах благодаря интерцепторам и удобной обработке ошибок.

Axios vs fetch

✅Автоматически парсит JSON
✅Бросает ошибку при 4xx/5xx автоматически
✅Интерцепторы — токен ко всем запросам
✅Прогресс загрузки файлов
✅axios.isAxiosError() — типизированные ошибки
❌fetch достаточен для простых проектов

Когда использовать

✅Нужны интерцепторы (авто-токен, авто-logout при 401)
✅Централизованная обработка ошибок
✅Загрузка файлов с прогресс-баром
✅Команда уже знает Axios
➡️Используется как queryFn внутри TanStack Query

Архитектура: один instance на весь проект

Компонент
→
api.get('/posts')
→
Request Interceptor (+ токен)
→
fetch https://api.com/posts
→
Response Interceptor (ошибки)
→
{ data }
Всегда создавай axios.create() instance вместо использования axios напрямую. Так baseURL и заголовки задаются один раз.

TanStack Query (React Query)

npm install @tanstack/react-query

Альтернатива RTK Query — управление серверными данными без Redux. Независим, гибче API, встроенный бесконечный скролл.

Когда TanStack Query, когда RTK Query

🔵RTK Query — уже используешь Redux в проекте
🔴TanStack Query — Redux не нужен, только серверные данные
🔴TanStack Query — нужен useInfiniteQuery из коробки
🔴TanStack Query — команда предпочитает его

Ключевые отличия от RTK Query

  • Не зависит от Redux — работает сам по себе
  • useInfiniteQuery — бесконечный скролл без доп. кода
  • queryKey как массив — гибкое кеширование
  • Query Factory паттерн — ключи и функции вместе
  • Лучше DevTools для отладки
  • Optimistic update через onMutate/onError/onSettled

queryKey — адрес в кеше

queryKeyЧто кешируется
['posts']Список всех постов
['posts', 42]Конкретный пост с id=42
['posts', { page: 1, sort: 'date' }]Посты с фильтрами — разные фильтры = разный кеш
['posts', postId, 'comments']Комментарии конкретного поста
Включай в queryKey всё от чего зависит запрос. При изменении любого элемента ключа — запрос перезапустится автоматически.

Tailwind CSS

npm install -D tailwindcss postcss autoprefixer

Utility-first CSS фреймворк. Вместо написания CSS-классов — применяешь готовые утилиты прямо в JSX. Итоговый бандл содержит только используемые классы.

Принцип работы

📌Классы описывают одно CSS-свойство: p-4 = padding: 16px
📌Адаптивность через префиксы: md:flex — flex от 768px
📌Состояния: hover:bg-blue-600, focus:ring-2
📌Шкала отступов: 1 = 4px, 2 = 8px, 4 = 16px, 8 = 32px

Когда использовать

✅Хочешь не переключаться между HTML и CSS файлами
✅Нужна быстрая адаптивность без медиазапросов вручную
✅Команда договорилась о дизайн-системе
➡️cn() = clsx + tailwind-merge — обязательная утилита

Error Boundaries

npm install react-error-boundary

Изолируют ошибки компонентов. Без них одна ошибка в любом компоненте роняет всё приложение — белый экран. С ними — только конкретный блок показывает fallback.

Что ловит Error Boundary

✅Ошибки в render()
✅Ошибки в lifecycle-методах
✅Ошибки в дочерних компонентах
❌Ошибки в event handlers → try/catch
❌Асинхронные ошибки (fetch, setTimeout) → try/catch

Стратегия расстановки

1️⃣Глобальный — оборачивает всё приложение (обязателен)
2️⃣По страницам — ошибка страницы не ломает навигацию
3️⃣По виджетам — упавший виджет не роняет дашборд
Чем мельче Error Boundary — тем меньше теряет пользователь при ошибке. Лучшая стратегия — все три уровня одновременно.

TypeScript

встроен в Vite при создании проекта

Добавляет статическую типизацию. Не выполняется в браузере — компилируется в JS. Вся польза во время разработки: IDE ловит ошибки до запуска кода.

Зачем в React-проекте

  • IDE знает какие props принимает компонент
  • Ошибка типа → до запуска, не в браузере
  • Рефакторинг безопаснее — TS найдёт все места
  • Типы Props = документация API компонента
  • Автодополнение полей, методов, импортов
  • Обязателен в командной разработке

Производительность — lazy, Suspense, Transitions

React 18 — встроено

Инструменты для быстрого UI: code splitting уменьшает начальный бандл, useTransition убирает подвисание при тяжёлых обновлениях, useDeferredValue делает поиск мгновенным.

lazy + Suspense

📌lazy() — загружает компонент только при первом переходе
📌Suspense — показывает fallback пока chunk грузится
✅Страницы — каждая страница отдельный chunk
✅Тяжёлые виджеты — независимая загрузка
✅С createBrowserRouter — loader и lazy параллельны

useTransitionuseDeferredValue
Что делаетПомечает setState как несрочноеЗадерживает обновление значения
КогдаКонтролируешь setStateЗначение приходит как prop/store
ПримерПереключение вкладокПоиск по загруженным данным
memo нуженНетДа — на компоненте-потребителе

Как lazy работает с createBrowserRouter — ключевое преимущество

BrowserRouter
→
chunk грузится → компонент рендерится → useEffect → fetch данных → skeleton → данные
createBrowserRouter
→
chunk + loader fetch параллельно → оба готовы → компонент сразу с данными
С createBrowserRouter loader и lazy работают параллельно — данные и код страницы грузятся одновременно. Пользователь ждёт меньше.

Жизненный цикл запроса в реальном приложении

1. Пользователь кликает /posts
→
2. Router: lazy chunk + loader параллельно
→
3. Axios: GET /api/posts (+ токен из Redux)
→
4. PostsPage рендерится с данными (useLoaderData)
→
5. Компонент читает user из Redux
→
6. Пользователь лайкает пост → useFetcher (без навигации)
→
7. mutation → invalidatesTags → список обновился
Главное правило: Router = URL и навигация. RTK Query/TanStack = данные от API. Redux/Zustand = клиентское состояние. useState = локальное состояние компонента. RHF = формы. Tailwind = стили. lazy = производительность.
